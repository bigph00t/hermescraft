**2. Extend handleUpdateTask to capture expected_outcome and trigger review:**
Replace the existing handleUpdateTask function (lines 229-257) with a version that:
- Accepts a 4th parameter: `expected_outcome` (optional string)
- When `status === 'done'` and `expected_outcome` is provided:
  - Instead of immediately marking done, set status to `'reviewing'`
  - Store `expected_outcome` on the subtask object
  - Set `pendingReview = { index, expected_outcome, reviewTick: tickCount + 1 }`
  - Return message: `"Subtask ${index} marked for review. Will verify: ${expected_outcome}"`
- When `status === 'done'` and NO expected_outcome: keep current behavior (immediate done + auto-advance)
- Add retry tracking fields: when a subtask enters review, ensure it has `retry_count: subtask.retry_count || 0` and `max_retries: 2`
- When `status === 'failed'`: increment `retry_count`, if `retry_count >= max_retries` set status to `'blocked'` instead and auto-advance, otherwise set status to `'in-progress'` (retry)
- Add the `expected_outcome` parameter to the update_task INFO_ACTION dispatch (line 705): `handleUpdateTask(response.action.index, response.action.status, response.action.note, response.action.expected_outcome)`

**3. Add reviewSubtaskOutcome function and call it at the start of tick:**
Create function `reviewSubtaskOutcome(state)` that:
- If `pendingReview` is null, returns null
- If `tickCount < pendingReview.reviewTick`, returns null (not time yet)
- Loads task state, gets the subtask at pendingReview.index
- If subtask status is not 'reviewing', clears pendingReview and returns null
- Performs a simple keyword-based check of expected_outcome against state:
  - If expected_outcome contains an item name (e.g., "have wooden_pickaxe"), check if that item is in state.inventory
  - If expected_outcome contains coordinates (e.g., "be at X,Y,Z"), check if position is within 5 blocks
  - If expected_outcome contains "health" with a number, check state.health
  - Default: if no keywords match, pass the review (trust the agent's judgment)
- On PASS: set subtask status to 'done', auto-advance next pending subtask, clear pendingReview, return `{ passed: true, subtaskIndex, expected_outcome }`
- On FAIL: set subtask status to 'failed' (which triggers retry logic — increment retry_count, set back to in-progress or blocked), clear pendingReview, return `{ passed: false, subtaskIndex, expected_outcome, actual: <what was found> }`
- Save task state after any status change

Call `reviewSubtaskOutcome(state)` in tick() right after the state is fetched (after line 357), store result in `let reviewResult = reviewSubtaskOutcome(state);`

Pass `reviewResult` to both buildUserMessage call sites as a new parameter.

  // Subtask review result — shown prominently so the agent sees what passed/failed
  if (reviewResult) {
    if (reviewResult.passed) {
      parts.push(`== REVIEW PASSED == Subtask ${reviewResult.subtaskIndex}: "${reviewResult.expected_outcome}" -- VERIFIED`)
    } else {
      parts.push(`== REVIEW FAILED == Subtask ${reviewResult.subtaskIndex}: expected "${reviewResult.expected_outcome}" but found: ${reviewResult.actual}`)

**3. Extend the subtask line rendering to show retry info:**
In the subtask rendering loop (lines 143-154), update the line construction to show retry count when > 0:
```javascript
  for (const st of taskProgress.subtasks) {
    let marker = '[ ]'
    if (st.status === 'done') marker = '[x]'
    else if (st.status === 'in-progress') marker = '[>]'
    else if (st.status === 'failed') marker = '[!]'
    else if (st.status === 'blocked') marker = '[B]'
    else if (st.status === 'reviewing') marker = '[?]'
    let line = `  ${st.index}. ${marker} ${st.text}`
    if (st.note) line += ` -- ${st.note}`
    if (st.retry_count > 0) line += ` (retry ${st.retry_count}/${st.max_retries || 2})`
    if (st.status === 'in-progress') line += '  <-- CURRENT'
    if (st.status === 'reviewing') line += '  <-- VERIFYING...'
    parts.push(line)
  }
```

This ensures the agent always sees: (a) whether its last subtask passed review, (b) which subtask is being verified, (c) how many retries have been used.
