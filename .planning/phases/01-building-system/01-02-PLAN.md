1. In HermesBridgeMod.java, add a method to capture the framebuffer as a PNG byte array. This must run on the render thread (client.execute or via scheduled tick):

   ```java
   private static byte[] lastScreenshot = null;
   private static long lastScreenshotTime = 0;
   private static final long SCREENSHOT_COOLDOWN_MS = 1000; // Min 1s between captures

   public static CompletableFuture<byte[]> captureScreenshot() {
       CompletableFuture<byte[]> future = new CompletableFuture<>();
       MinecraftClient client = MinecraftClient.getInstance();
       if (client == null || client.getFramebuffer() == null) {
           future.complete(null);
           return future;
       }
       // Must execute on render thread
       client.execute(() -> {
           try {
               Framebuffer fb = client.getFramebuffer();
               NativeImage image = ScreenshotRecorder.takeScreenshot(fb);
               // Scale down to 640x360 for speed (per decision D-04)
               // NativeImage doesn't have easy resize, so we'll use full res
               // and let the consumer handle it — or crop center
               ByteArrayOutputStream baos = new ByteArrayOutputStream();
               image.writeTo(baos);  // writes PNG
               image.close();
               future.complete(baos.toByteArray());
           } catch (Exception e) {
               LOGGER.error("[HermesBridge] Screenshot capture failed", e);
               future.complete(null);
           }
       });
       return future;
   }
   ```

   Required imports in HermesBridgeMod.java:
   - `net.minecraft.client.gl.Framebuffer`
   - `net.minecraft.client.texture.NativeImage`
   - `net.minecraft.client.util.ScreenshotRecorder`
   - `java.io.ByteArrayOutputStream`
   - `java.util.concurrent.CompletableFuture`

   Note: Check the MC 1.21.1 Fabric API — `ScreenshotRecorder.takeScreenshot` takes a `Framebuffer` and returns `NativeImage`. The NativeImage can be written as PNG via `writeTo(Path)` or `writeTo(OutputStream)`. Verify exact method signatures against the access widener. If `writeTo(OutputStream)` is not available, use `writeTo(Path)` to a temp file and read the bytes.

2. In HttpServer.java, add a new ScreenshotHandler and register it:

   In constructor, add:
   ```java
   server.createContext("/screenshot", new ScreenshotHandler());
   ```

   Handler:
   ```java
   static class ScreenshotHandler implements HttpHandler {
       @Override
       public void handle(HttpExchange exchange) throws IOException {
           if ("OPTIONS".equalsIgnoreCase(exchange.getRequestMethod())) {
               sendResponse(exchange, 204, "");
               return;
           }
           try {
               byte[] png = HermesBridgeMod.captureScreenshot().get(5, java.util.concurrent.TimeUnit.SECONDS);
               if (png == null || png.length == 0) {
                   sendResponse(exchange, 500, "{\"error\":\"Screenshot capture failed\"}");
                   return;
               }
               exchange.getResponseHeaders().set("Content-Type", "image/png");
               exchange.getResponseHeaders().set("Access-Control-Allow-Origin", "*");
               exchange.sendResponseHeaders(200, png.length);
               try (OutputStream os = exchange.getResponseBody()) {
                   os.write(png);
               }
           } catch (Exception e) {
               HermesBridgeMod.LOGGER.error("[HermesBridge] Screenshot endpoint error", e);
               sendResponse(exchange, 500, "{\"error\":\"Screenshot failed: " + e.getMessage() + "\"}");
           }
       }
   }
   ```

   Add import for `java.util.concurrent.TimeUnit` if not already present.

3. Verify the access widener (`mod/src/main/resources/hermescraft.accesswidener`) allows access to Framebuffer and ScreenshotRecorder. If not, add the necessary entries. The ScreenshotRecorder.takeScreenshot method may need widening.

2. Create `agent/vision.js` — the vision loop module. Follow project conventions: camelCase exports, ES modules, one-line header comment, no default exports.

   Module state:
   - `_visionInterval` — setInterval handle
   - `_lastVisionText` — cached vision context string
   - `_running` — boolean

   Exports:
   - `startVisionLoop(agentConfig)` — starts the async loop that runs every VISION_INTERVAL_MS:
     a. Fetch screenshot: `const res = await fetch(MOD_URL + '/screenshot', { signal: AbortSignal.timeout(8000) })`
     b. Convert response to base64: `const buf = await res.arrayBuffer(); const b64 = Buffer.from(buf).toString('base64')`
     c. Call LLM with vision prompt using the OpenAI client (create a local client instance or import from llm.js):
        ```javascript
        const visionRes = await client.chat.completions.create({
          model: VISION_MODEL,
          messages: [{
            role: 'user',
            content: [
              { type: 'image_url', image_url: { url: `data:image/png;base64,${b64}` } },
              { type: 'text', text: 'Describe what you see in this Minecraft screenshot. Note: terrain features (water, cliffs, caves, flat areas), nearby structures or builds, mobs and animals, other players, resources (trees, ores), and any hazards. Be concise — 3-5 sentences max.' }
            ]
          }],
          max_tokens: 200,
          temperature: 0.3,
        })
        ```
     d. Extract the text response
     e. Write to `{dataDir}/vision-context.txt` with timestamp header
     f. Update `_lastVisionText`
     g. Wrap entire loop body in try/catch — vision failures are non-fatal, just log and continue

   - `stopVisionLoop()` — clears the interval, sets _running = false
   - `getVisionContext()` — returns _lastVisionText (for direct access without file I/O)

   The vision-context.txt format:
   ```
   == WHAT I SEE (updated {seconds}s ago) ==
   {vision description from LLM}
   ```

   Error handling:
   - If screenshot fetch fails (mod not running, timeout), log warning and skip
   - If LLM vision call fails, log warning and keep last known vision
   - Never crash the agent process — vision is advisory, not critical
