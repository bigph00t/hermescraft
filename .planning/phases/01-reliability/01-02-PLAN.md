    **Part A — Chat dedup hardening (COM-01) in agent/index.js:**

    The current dedup (lines 41, 393-398) uses a hash of all visible messages. This breaks when:
    - Messages cycle out of the 20-message ring buffer (old messages disappear, making the hash change even if no new messages arrived, but more importantly old messages that re-enter look new)
    - Same messages in different order produce different hashes

    Replace the string-hash approach with a monotonic message-count tracker:

    1. Replace line 41:
       ```javascript
       let lastSeenChatHash = '';
       ```
       With:
       ```javascript
       let lastProcessedMessages = new Set()
       ```

    2. Replace the dedup block (lines 393-398). The current code is:
       ```javascript
       const chatHash = playerMessages.join('|');
       if (playerMessages.length > 0 && chatHash !== lastSeenChatHash) {
         const oldMessages = lastSeenChatHash.split('|');
         const newMessages = playerMessages.filter(m => !oldMessages.includes(m));
         lastSeenChatHash = chatHash;
       ```

       Replace with:
       ```javascript
       if (playerMessages.length > 0) {
         // Create unique-ish keys: message text + position in batch
         // This handles repeated identical messages correctly
         const currentBatch = new Set(playerMessages.map((m, i) => `${i}:${m}`))
         const newMessages = playerMessages.filter((m, i) => !lastProcessedMessages.has(`${i}:${m}`))
         lastProcessedMessages = currentBatch
       ```

       The rest of the block (lines 400-433) stays unchanged — it already uses `newMessages`.

    **Part B — AutoConnect reset on disconnect (COM-02) in HermesBridgeMod.java:**

    The bug: `autoConnectAttempted` is set to `true` before the connect call (line 93) and only reset in the catch block (line 101). If the player connects successfully and then gets kicked/disconnected, `autoConnectAttempted` remains `true` forever, preventing reconnection.

    Fix: Track whether the player was previously connected. When the player transitions from connected (non-null) to disconnected (null), reset `autoConnectAttempted`.

    1. Add a new field after line 27 (`private boolean autoConnectAttempted = false;`):
       ```java
       private boolean wasConnected = false;
       ```

    2. Inside the `ClientTickEvents.END_CLIENT_TICK.register` lambda (line 84), add player disconnect detection BEFORE the auto-connect block (before line 88). Insert after `tickCounter++;` (line 85):
       ```java
       // Detect player disconnect — reset auto-connect flag so we can reconnect
       if (wasConnected && client.player == null) {
           LOGGER.info("[HermesBridge] Player disconnected — will auto-reconnect");
           autoConnectAttempted = false;
       }
       wasConnected = client.player != null;
       ```

    This ensures that whenever the player transitions from connected to disconnected (kick, timeout, manual disconnect), the auto-connect logic is re-enabled.
  </action>
