Create `mind/buildPlanner.js` following the module header convention (`// buildPlanner.js -- Purpose`) and the established init pattern.

**Module state:**
- `_buildsDir` — path to `data/<agent>/builds/`, set by `initBuildPlanner(config)`

**Exported functions (10 total):**

1. `initBuildPlanner(config)` — create `_buildsDir = join(config.dataDir, 'builds')`, mkdirSync if needed. Follow same pattern as `initBuildHistory(config)`.

2. `saveBuildPlan(plan)` — atomic write: writeFileSync to `.tmp`, renameSync to final path `<_buildsDir>/<plan.id>.json`. Follow `build-history.js` write pattern.

3. `loadBuildPlan(planId)` — read + JSON.parse from `<_buildsDir>/<planId>.json`, return null on not-found or parse error.

4. `listBuildPlans()` — readdirSync _buildsDir, filter `.json` files excluding `-section-` files, parse each, return array. Return `[]` if dir missing.

5. `getActivePlan()` — `listBuildPlans().find(p => p.status === 'active') || null`

6. `decomposeSections(spec)` — deterministic section layout from spec `{ dimensions: { width, height, depth }, style, materials }`. Returns array of section objects:
   - `{ id: 'foundation', offsetX: 0, offsetY: 0, offsetZ: 0, maxW: width, maxD: depth, maxH: 1 }`
   - `{ id: 'north_wall', offsetX: 0, offsetY: 1, offsetZ: 0, maxW: width, maxD: 1, maxH: height - 2 }`
   - `{ id: 'south_wall', offsetX: 0, offsetY: 1, offsetZ: depth - 1, maxW: width, maxD: 1, maxH: height - 2 }`
   - `{ id: 'east_wall', offsetX: width - 1, offsetY: 1, offsetZ: 1, maxW: 1, maxD: depth - 2, maxH: height - 2 }`
   - `{ id: 'west_wall', offsetX: 0, offsetY: 1, offsetZ: 1, maxW: 1, maxD: depth - 2, maxH: height - 2 }`
   - `{ id: 'roof', offsetX: 0, offsetY: height - 1, offsetZ: 0, maxW: width, maxD: depth, maxH: 1 }`
   - If any section dimension exceeds 10 on any axis, further split it (e.g., north_wall 20 wide -> north_wall_a and north_wall_b each 10 wide). This ensures each section fits the 10x10x8 blueprint constraint.

7. `auditMaterials(inventory, plan)` — count all blocks needed across pending sections by loading each section blueprint JSON, resolving palette characters to block names (use same resolution logic as `build.js` lines 157-168: first preferred block found in inventory, else first preferred, else 'cobblestone'). Compare against `inventory` (array of `{ name, count }`). Return `{ needed: {block: count}, have: {block: count}, gap: {block: count}, ready: boolean }`. Update `plan.materialAudit` in place. Note: `inventory` is passed as a simple array, not a bot object (keeps function testable without mineflayer).

8. `planBuild(description, queryLLMFn, buildDesignPromptFn, validateBlueprintFn, botPosition)` — async function:
   - Generate plan ID via `crypto.randomUUID().slice(0, 8)`
   - Step 1: Call `queryLLMFn(buildSpecPrompt(description), 'Generate the build spec JSON now.')` to get overall spec (style, dimensions, materials)
   - Step 2: Extract JSON from response (strip `<think>` tags, regex `/{[\s\S]*}/`, parse)
   - Step 3: Call `decomposeSections(spec)` to get section list
   - Step 4: For each section, call `queryLLMFn(buildDesignPromptFn(sectionDescription, []), 'Generate the section blueprint JSON now.')`, validate with `validateBlueprintFn`, write to `<_buildsDir>/<planId>-section-<sectionId>.json`
   - Step 5: Assemble plan object with schema from RESEARCH.md Pattern 2 (id, description, createdAt, origin from botPosition, status: 'active', spec, materialAudit: null, sections array, totalEstimatedBlocks)
   - Step 6: `saveBuildPlan(plan)`, return `{ success: true, planId, sections, totalBlocks }`
   - On any failure: return `{ success: false, reason: '...' }`

9. `getBuildPlanForPrompt()` — get active plan, format progress string. If material gap exists, show "MISSING MATERIALS: stone_bricks*336, oak_planks*120". Otherwise show "Next: <section_id>. Use !build to continue." Return empty string if no active plan.

10. `buildExpectedBlockMap(blueprint, originX, originY, originZ)` — given a blueprint JSON and origin coords, build a Map<string, string> where key is `"x,y,z"` and value is the resolved block name. Used by Plan 02 for post-build diff. Iterate layers/grid same as `build.js` queue building (lines 170-190), using first preferred block from palette.

**Internal helpers (not exported):**

- `buildSpecPrompt(description)` — system prompt for the spec-generation LLM call. Ask for JSON with `{ style, materials: { primary, secondary, accent }, dimensions: { width, height, depth } }`. Constrain: max 30x30 footprint, max 15 tall. No coordinate arrays.
- `extractJSON(raw)` — strip `<think>` blocks, try direct parse, then regex `/{[\s\S]*}/` fallback. Return parsed object or null.
- `countBlocks(blueprint)` — count non-'.' non-' ' characters across all layer grids.

**Schema adherence:** The plan JSON must match the schema in RESEARCH.md Pattern 2 exactly. Each section in the `sections` array has: `id, status, blueprintFile, completedIndex (0), blockCount, claimedBy (null), repairAttempts (0)`.

**Anti-patterns to avoid (from RESEARCH.md):**
- Do NOT write section blueprints to `body/blueprints/` — always use `_buildsDir`
- Do NOT ask the LLM for coordinate arrays — only style/dimensions/materials
- Do NOT count palette characters without resolving them first in `auditMaterials()`
