Create `mind/llm.js` — the LLM client with conversation memory and !command parsing. Port patterns from `agent/llm.js` but adapted for the new Mind architecture:

**mind/llm.js must contain:**

1. **OpenAI client setup** — identical to agent/llm.js lines 5-28:
   - `import OpenAI from 'openai'`
   - `baseURL` from `process.env.VLLM_URL || 'http://localhost:8000/v1'`
   - `apiKey` from `process.env.VLLM_API_KEY || 'not-needed'`
   - `timeout: 120000`
   - Model from `process.env.MODEL_NAME || 'MiniMaxAI/MiniMax-M2.7'` (note: default changed from Hermes to MiniMax)
   - Temperature from `process.env.TEMPERATURE || '0.6'`
   - NO OAuth detection needed (v2 uses direct vLLM, not Anthropic)

2. **Conversation history** — module-level array `conversationHistory`:
   - `clearConversation()` — resets to empty array
   - `getHistory()` — returns the array (read-only reference)
   - `MAX_HISTORY_MESSAGES = 80` (40 turns, matching MIND-03 requirement)
   - `trimHistory()` — removes oldest complete rounds (user+assistant+optional tool) from front when over cap. Port from agent/llm.js lines 55-70 verbatim.
   - `trimHistoryGraduated(fraction = 0.25)` — removes oldest `fraction` of messages, round-boundary-aware. Port from agent/llm.js lines 72-105 verbatim.

3. **queryLLM(systemPrompt, userMessage)** — sends prompt to LLM, returns `{ reasoning, command, args, raw }`:
   - Builds messages array: `[{ role: 'system', content: systemPrompt }, ...conversationHistory, { role: 'user', content: userMessage }]`
   - Calls `client.chat.completions.create({ model, messages, temperature })` — TEXT MODE ONLY, no tools/tool_choice (per research: MiniMax M2.7 !command text is the reliable path)
   - Extracts `msg.content` from `response.choices[0].message`
   - Strips `<think>...</think>` tags: `content.replace(/<think>[\s\S]*?<\/think>/g, '').trim()`
   - Runs `parseCommand(strippedText)` to extract `{ command, args }` or null
   - If parseCommand returns null, result is `{ reasoning: strippedText, command: null, args: {}, raw: msg.content }`
   - Pushes `{ role: 'user', content: userMessage }` and `{ role: 'assistant', content: msg.content }` to conversationHistory
   - Calls `trimHistory()` after push
   - On context overflow error (detect via `isContextOverflowError` — same patterns as agent/llm.js lines 138-144), call `trimHistoryGraduated(0.25)` and retry (up to 3 retries with exponential backoff)
   - On all retries exhausted, call `trimHistoryGraduated(0.5)` then throw
   - Export: `export async function queryLLM(systemPrompt, userMessage)`

4. **parseCommand(text)** — extracts `!command` from text:
   - Regex: `text.match(/!(\w+)(?:\s+(.*))?/)`
   - Named args: `argStr.matchAll(/(\w+):("([^"]*?)"|(\S+))/g)` → key:value pairs, strip quotes
   - Positional fallback: if no named args, split on whitespace → `{ item: parts[0], count: parseInt(parts[1]) }` if parts exist
   - For `!chat`, special handling: everything after `message:` is the message content, including quoted strings
   - Returns `{ command: name, args }` or `null`
   - NOT exported (internal helper)

5. **isContextOverflowError(err)** — same logic as agent/llm.js lines 138-144. NOT exported.

**mind/prompt.js must contain:**

1. **buildSystemPrompt(bot, options = {})** — minimal system prompt for Phase 3:
   - Identity section: hardcoded persona text (simple for now — Phase 5 adds SOUL files):
     ```
     You are {bot.username}, a Minecraft player. You observe the world and take actions using !commands.
     ```
   - Command reference section listing all available !commands with argument syntax (see research Pattern 3 for the exact format)
   - Available commands: `!gather item:name count:N`, `!mine item:name count:N`, `!craft item:name count:N`, `!smelt item:name fuel:name count:N`, `!navigate x:N y:N z:N`, `!chat message:"text"`, `!idle`
   - Few-shot examples section (3-4 examples per research Pitfall 5)
   - Format instruction: "Your response must contain exactly one !command line."
   - `options.soul` — optional string prepended as identity (for future Phase 5 use)
   - Export: `export function buildSystemPrompt(bot, options = {})`

2. **buildStateText(bot)** — compact game state summary:
   - Position: `pos: X,Y,Z` from `bot.entity?.position` (Math.round each)
   - Health/food: `health: N/20  food: N/20` from `bot.health`, `bot.food`
   - Inventory: `inventory: Nx item1, Nx item2` from `bot.inventory.items().map(i => \`${i.count}x ${i.name}\`).join(', ')` or `'empty'`
   - Time: `time: day/night` from `bot.time.timeOfDay` (0-12000 = day, 12000-24000 = night)
   - Nearby entities: count of players and hostile mobs within 16 blocks
   - Export: `export function buildStateText(bot)`

3. **buildUserMessage(bot, trigger, options = {})** — user message for LLM:
   - Starts with trigger context: `[chat from {sender}: {message}]` or `[skill complete: {skillName} — {result}]` or `[idle for {idleMs}ms — what should I do?]`
   - Appends `buildStateText(bot)`
   - `options.skillResult` — the `{ success, reason }` from the last skill (for skill_complete trigger)
   - `options.sender` / `options.message` — for chat trigger
   - `options.idleMs` — for idle trigger
   - Export: `export function buildUserMessage(bot, trigger, options = {})`

Both files follow project conventions: ES Modules, 2-space indent, no-semi, single quotes, named exports only, one-line module header comment.

**mind/registry.js must contain:**

1. **Imports from body/**:
   ```javascript
   import { gather } from '../body/skills/gather.js'
   import { mine } from '../body/skills/mine.js'
   import { craft } from '../body/skills/craft.js'
   import { smelt } from '../body/skills/smelt.js'
   import { navigateTo } from '../body/navigate.js'
   import { requestInterrupt, clearInterrupt } from '../body/interrupt.js'
   ```

2. **REGISTRY Map** — maps command name strings to async handler functions:
   ```javascript
   const REGISTRY = new Map([
     ['gather',   (bot, args) => gather(bot, args.item, parseInt(args.count) || 1)],
     ['mine',     (bot, args) => mine(bot, args.item, parseInt(args.count) || 1)],
     ['craft',    (bot, args) => craft(bot, args.item, parseInt(args.count) || 1)],
     ['smelt',    (bot, args) => smelt(bot, args.item, args.fuel || 'coal', parseInt(args.count) || 1)],
     ['navigate', (bot, args) => navigateTo(bot, parseInt(args.x), parseInt(args.y), parseInt(args.z))],
     ['chat',     (bot, args) => { bot.chat(args.message || ''); return { success: true } }],
     ['idle',     (_bot, _args) => Promise.resolve({ success: true, reason: 'idle' })],
   ])
   ```

3. **dispatch(bot, command, args)** — the main export:
   - Looks up `command` in REGISTRY
   - If not found, returns `{ success: false, reason: \`unknown command: !${command}\` }`
   - If found:
     a. Calls `requestInterrupt(bot)` to cancel any in-flight skill
     b. Calls `clearInterrupt(bot)` to reset for the new skill
     c. Calls `const result = await fn(bot, args)`
     d. Returns `result`
   - Wrapped in try/catch — on error returns `{ success: false, reason: err.message }`
   - Export: `export async function dispatch(bot, command, args)`

4. **listCommands()** — returns array of command name strings from REGISTRY.keys():
   - Export: `export function listCommands()`

The registry does NOT track skillRunning state — that is managed by mind/index.js (Plan 02) which wraps dispatch calls. This keeps the registry as a pure lookup + dispatch bridge.

All args values come as strings from parseCommand — the registry must `parseInt()` numeric args (count, x, y, z). The `args.item` is passed through as-is since body/ skills handle normalization internally.

File follows project conventions: ES Modules, 2-space indent, no-semi, single quotes, named exports only, one-line module header comment `// registry.js -- Command name to body/ skill dispatch bridge`.
