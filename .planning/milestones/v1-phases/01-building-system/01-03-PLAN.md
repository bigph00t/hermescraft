<objective>
Create the blueprint executor engine and wire it as a sustained `build` tool. The executor reads a blueprint, resolves materials from inventory, and places blocks coordinate-by-coordinate via the mod's place action. It runs as a sustained action across multiple ticks (like mine/navigate), reporting progress back to the LLM.

Purpose: This is the core building mechanic. The LLM says "build small-cabin at x,y,z", and procedural code handles all the spatial math (anti-pattern #1: never let the LLM calculate coordinates). Decisions D-11 through D-14 from CONTEXT.md.

Output: builder.js module, build tool in tools.js, action integration.

  <action>
Create `agent/builder.js` — the blueprint execution engine. Follow project conventions: camelCase exports, ES modules, one-line header comment, no default exports.

Module state:
- `_activeBuild` — object tracking current build or null:
  ```javascript
  {
    blueprintName: string,
    blueprint: object,        // loaded blueprint data
    origin: { x, y, z },     // world coordinates of blueprint origin
    palette: object,          // resolved palette mapping
    totalBlocks: number,      // total non-air blocks
    placedBlocks: number,     // successfully placed count
    queue: [],                // remaining blocks to place: [{x,y,z,block}, ...]
    startedAt: number,        // Date.now()
    paused: boolean,          // true when missing materials
    missingMaterials: [],     // what's needed to continue
  }
  ```

Exports:

1. `startBuild(blueprintName, originX, originY, originZ, inventory)`:
   - Load blueprint via `loadBlueprint(blueprintName)`
   - If blueprint not found, return `{ success: false, error: 'Unknown blueprint' }`
   - Extract available block names from inventory array (items the agent has)
   - Resolve palette via `resolvePalette(blueprint, availableBlocks)`
   - Build the block queue: iterate layers bottom-to-top (y=0 first), then row by row (z), then column (x). For each non-air character in the grid, push `{ x: originX + col, y: originY + layer.y, z: originZ + row, block: palette[char] }` to the queue
   - Set `_activeBuild` with the computed state
   - Return `{ success: true, message: 'Started building {name}. {totalBlocks} blocks to place.', totalBlocks, materials: Object.fromEntries(getBlueprintMaterials(blueprint)) }`

2. `resumeBuild(inventory)` — called each tick when build is active:
   - If no active build or paused, return `{ active: false }`
   - Check what blocks the agent currently has in inventory
   - Place up to 3 blocks per tick call (rate limit to avoid overwhelming the mod):
     For each block in queue (shift from front):
     a. Check if agent has the required block in inventory
     b. If not, pause the build: set `paused: true`, compute `missingMaterials` list, return progress with `{ active: true, paused: true, missingMaterials: [...], placed: N, total: M }`
     c. Send place action via fetch: `POST ${MOD_URL}/action` with body `{ "type": "place", "item": block.block, "x": block.x, "y": block.y, "z": block.z }`
     d. Increment placedBlocks
     e. If queue is empty, clear `_activeBuild`, return `{ active: false, complete: true, placed: totalBlocks }`
   - Return progress: `{ active: true, paused: false, placed: placedBlocks, total: totalBlocks, percent: Math.round(100 * placedBlocks / totalBlocks) }`

3. `getBuildProgress()` — returns current build state summary string for prompt injection:
   - If no active build: return ''
   - If paused: return `"BUILD PAUSED: {name} ({placed}/{total} blocks). Need: {missingMaterials.join(', ')}. Gather materials and I'll resume automatically."`
   - If active: return `"BUILDING: {name} ({placed}/{total} blocks, {percent}%). Do not interrupt."`

4. `cancelBuild()` — cancels active build, returns summary of what was placed

5. `isBuildActive()` — returns boolean

6. `unpauseBuild()` — sets paused=false so next resumeBuild attempts to continue

Important implementation details:
- The `resumeBuild` function must be async (it makes HTTP calls to the mod)
- Use `AbortSignal.timeout(3000)` on place calls to prevent hangs
- If a place call fails, don't remove the block from queue — retry next tick
- Log progress via the project's logger pattern (import from logger.js if needed, or just console.log with prefix)
- MOD_URL from `process.env.MOD_URL || 'http://localhost:3001'`
- When checking inventory for a block, handle `minecraft:` prefix: strip it from both sides before comparing
- The `inventory` parameter is the array from game state: `[{item: "minecraft:oak_planks", count: 32}, ...]`
- Before placing, check if agent is within ~8 blocks of target position. If too far, return a message telling the agent to navigate closer first. Use the origin coordinates for this check.

   ```javascript
   {
     type: 'function',
     function: {
       name: 'build',
       description: 'Build a structure from blueprints. Available: small-cabin, animal-pen, crop-farm. Specify where to place the origin corner.',
       parameters: {
         type: 'object',
         properties: {
           blueprint: { type: 'string', description: 'Blueprint name: "small-cabin", "animal-pen", "crop-farm"' },
           x: { type: 'integer', description: 'Origin X coordinate (front-left corner)' },
           y: { type: 'integer', description: 'Origin Y coordinate (ground level)' },
           z: { type: 'integer', description: 'Origin Z coordinate (front-left corner)' },
         },
         required: ['blueprint', 'x', 'y', 'z'],
       },
     },
   },
   ```

   Also add a `cancel_build` tool:
   ```javascript
   {
     type: 'function',
     function: {
       name: 'cancel_build',
       description: 'Cancel the current building project.',
       parameters: { type: 'object', properties: {} },
     },
   },
   ```

2. In agent/actions.js:
   - Add 'build' and 'cancel_build' to VALID_ACTIONS Set
   - Add schema validators:
     ```javascript
     build: (a) => typeof a.blueprint === 'string' && a.x !== undefined && a.y !== undefined && a.z !== undefined,
     cancel_build: () => true,
     ```
   - Do NOT add to INFO_ACTIONS — build is a game-world action

3. In agent/index.js, integrate the build system into the tick loop:

   a. Add import at top:
      ```javascript
      import { startBuild, resumeBuild, getBuildProgress, cancelBuild, isBuildActive, unpauseBuild } from './builder.js'
      ```

   b. Add 'build' to the SUSTAINED_ACTIONS Set (existing: mine, navigate, break_block)

   c. In the action dispatch section of tick() (where actions are executed), add handling for 'build':
      ```javascript
      case 'build': {
        const state = await fetchState()
        const result = startBuild(action.blueprint, action.x, action.y, action.z, state.inventory || [])
        // return result as action feedback
        break
      }
      case 'cancel_build': {
        const result = cancelBuild()
        break
      }
      ```

   d. At the START of each tick (before LLM query), check if a build is active:
      ```javascript
      if (isBuildActive()) {
        const state = await fetchState()
        const buildResult = await resumeBuild(state.inventory || [])
        // If build is paused for materials, the progress message will tell the LLM
        // If build completed, it will report completion
      }
      ```

   e. Inject build progress into the user message. In the section where buildUserMessage is called, add build progress to the state summary or as a separate section. The simplest approach: append `getBuildProgress()` to the state summary string if non-empty.

   f. When build is paused and the agent gathers more materials, the build should auto-unpause. In the resumeBuild flow, if the build was paused but now the agent has materials, call `unpauseBuild()` and continue. This should be handled inside builder.js's resumeBuild — if paused, re-check inventory and unpause if materials are now available.

Note: The build action handler needs to be added to the existing action dispatch logic in index.js. Look for the switch/if-else chain that handles different action types (around the executeAction call or the info action handlers). The `build` action should be handled BEFORE the generic `executeAction` call since it's managed by builder.js, not the mod API.
