  <action>
Create agent/crafter.js with the following structure:

**Header comment:** `// crafter.js — BFS recipe chain solver using minecraft-data 1.21.1`

**Imports:**
```javascript
import minecraftData from 'minecraft-data'
import { normalizeItemName } from './normalizer.js'
```

**Module state:**
- `let mcData = null` — loaded once by initCrafter()

**initCrafter() export:**
- Call `mcData = minecraftData('1.21.1')`
- Log: `[Crafter] Loaded ${Object.keys(mcData.recipes).length} recipe types`
- No arguments needed — this module has no per-agent state

**needsCraftingTable(recipe) helper (not exported):**
- If recipe has `inShape`: check if any row has length >= 3 OR if there are >= 3 rows → return true
- If shapeless (`ingredients` array): if ingredients.length > 4 → return true (but this is rare in vanilla MC)
- Otherwise return false

**selectBestVariant(recipes, inventory) helper (not exported):**
- `recipes` is an array of recipe variants for the same output item
- `inventory` is an object like `{ oak_planks: 10, stick: 4 }`
- For each recipe variant, extract all unique ingredient IDs, resolve to names via `mcData.items[id].name`
- Score each variant: count how many distinct ingredient types the inventory already has
- Return the variant with the highest score (most ingredients already available)
- Tie-break: prefer the variant at the END of the array (oak variants are last, which are the most common early-game)

**getIngredients(recipe) helper (not exported):**
- If recipe has `inShape`: flatten the 2D array, filter out nulls, count occurrences of each ID
- If shapeless (`ingredients`): count occurrences of each ID
- Return array of `{ item: string (name from mcData.items), count: number }`
- Call `normalizeItemName()` on each resolved name

**solveCraft(targetItem, inventory) export:**
- `targetItem` is a string like 'wooden_pickaxe'
- `inventory` is an object `{ itemName: count, ... }` — keys are normalized item names, values are counts
- Normalize targetItem via `normalizeItemName(targetItem)`
- Look up `mcData.itemsByName[normalizedTarget]` — if not found, return `{ steps: [], missing: [targetItem] }`
- Look up `mcData.recipes[item.id]` — if no recipes, return `{ steps: [], missing: [targetItem] }` (it's a raw material)
- BFS/DFS approach: recursively resolve each ingredient:
  1. If inventory has enough of ingredient → skip (no craft needed)
  2. If ingredient has no recipe → add to `missing` list
  3. If ingredient has recipe → recursively solve, then add craft step
- Build steps in dependency order (leaves first, target last)
- Each step: `{ action: 'craft', item: string, count: number, ingredients: [{item, count}], needsTable: boolean }`
- `count` is how many times to call craft (adjusted for recipe output count — e.g. planks recipe yields 4, so crafting planks once from 1 log gives 4)
- Use `selectBestVariant()` for recipe variant selection
- Track visited items to prevent infinite recursion on circular recipes (shouldn't exist in vanilla but be safe)
- Return `{ steps: [...], missing: [...] }`

**Quantity math:**
- When target needs N of an ingredient, and the recipe produces `result.count` per craft: need `Math.ceil(N / result.count)` craft operations
- Excess production is tracked so downstream steps don't re-craft unnecessarily
- Example: wooden_pickaxe needs 3 planks + 2 sticks. Planks recipe yields 4 per log. So 1 craft of planks gives 4 (enough for the 3 needed for the pickaxe). Sticks recipe yields 4 per 2 planks. So 1 craft of sticks gives 4 (enough for the 2 needed). Total: 1 log needed.

**Do NOT:**
- Do not create a knowledge/recipes.json file — use minecraft-data directly (per user decision: load at startup, not maintain a separate file)
- Do not make any HTTP calls to the mod's /recipes endpoint
- Do not modify any other files in this task
