Extend agent/locations.js with the following additions (keep all existing exports unchanged):

1. **New `resources` section in locations data.** The `locations` module-level variable currently stores flat key-value pairs. Add a parallel `resources` object that is loaded from and saved to the same `locations.json` file. Structure:
```javascript
// In initLocations, after loading locations.json:
// If the loaded JSON has a 'resources' key, extract it. Otherwise default to {}.
// The file format becomes: { ...regular locations..., _resources: { ... } }
// Use _resources key to avoid collision with a location named "resources".
```
Each resource entry: `{ name, type, x, y, z, dimension, timestamp, metadata }` where:
- `type` is one of: `'ore_vein'`, `'tree_cluster'`, `'build_site'`, `'chest'`, `'poi'`
- `dimension` defaults to `'overworld'`
- `metadata` is an optional object (e.g., `{ blockType: 'iron_ore', count: 5 }`)

2. **New export `saveResourcePatch(name, type, x, y, z, metadata = {})`:**
- Validates `type` against the allowed set. If invalid, defaults to `'poi'`.
- Stores in the `resources` object keyed by name.
- Calls `pruneResources()` if total entries exceed 500.
- Calls `saveLocations()` to persist.

3. **New function `pruneResources()`:**
- When total resource entries exceed 500, find entries of the same type within 10 blocks of each other.
- Keep the newest entry, delete the older duplicate.
- Continue until under 500 or no more duplicates found.

4. **New export `getResourcesForPrompt(position)`:**
- Accepts the agent's current position `{ x, y, z }`.
- Filters to entries within 150 blocks (Euclidean distance, skip y for horizontal distance: `sqrt(dx*dx + dz*dz)`).
- Sorts by distance ascending.
- Caps at 20 entries.
- Returns formatted string: `'Nearby resources: oak_cluster_1 (ore_vein, 45 blocks N), iron_vein_3 (ore_vein, 80 blocks E), ...'`
- Direction computed as cardinal (N/S/E/W) based on dx/dz dominance.
- Returns empty string if no entries within range.

5. **Modify `getLocationsForPrompt(position)` (BREAKING CHANGE — now accepts position parameter):**
- Add optional `position` parameter.
- If position provided: filter to entries within 150 blocks, sort by distance, cap at 10 entries.
- If position is null/undefined: fall back to current behavior (return all — backward compat for any callers not updated yet).
- Format stays the same: `'Known locations: home: (10, 64, 20), storage: (15, 64, 22)'`

6. **Modify `saveLocations()`:**
- When writing to disk, merge the `resources` object into the JSON under `_resources` key.
- When reading in `initLocations`, extract `_resources` back out and delete it from the main `locations` object.

7. **Modify `getChestsForPrompt(position)` in agent/chests.js (BREAKING CHANGE — now accepts position parameter):**
- Add optional `position` parameter.
- If position provided: parse chest key `"x,y,z"` to coordinates, filter to within 150 blocks, sort by distance, cap at 10 entries.
- If position is null/undefined: fall back to current behavior (return all).
- Format stays the same.

8. **Auto-detection hook in `autoDetectLocations(state)`:**
- After the existing auto-detection logic, scan `state.surfaceBlocks` (if present) for ore blocks. If any ore block type (iron_ore, coal_ore, gold_ore, diamond_ore, copper_ore, lapis_ore, redstone_ore, emerald_ore) appears 2+ times, call `saveResourcePatch()` with type `'ore_vein'` and metadata `{ blockType, count }`. Use a generated name like `'iron_vein_1'` (increment based on existing entries of same blockType).
- For tree clusters: if 3+ log blocks of same type are found, save as `'tree_cluster'` with generated name.
- Deduplicate: skip saving if an entry of the same type already exists within 15 blocks.
